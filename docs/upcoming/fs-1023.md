# FS-1023 — Type Providers Generate Types From Types

Date: 2025-11-12  
Status: In development / dogfooding

## Overview

FS-1023 extends generative type providers so they can accept an existing `System.Type` as a static argument and emit a new concrete type that mirrors the source shape. Instead of inlining provider-supplied IL, the compiler now captures each generated member inside the typed tree (`ProvidedMemberBinding`) and emits the final IL itself. This unlocks relocation (e.g., `/standalone` builds) and gives downstream tooling—IDE, reflection, scripting—a stable view of the generated surface.

Key goals:

- Preserve the identity of the `System.Type` static argument through project system reloads, incremental builds, and reflection.
- Publish provider-generated members into the typed tree (`tcaug_adhoc_list`) so consumers (IlxGen, InfoReader, reflection builders) no longer query the provider at call sites.
- Emit IL for provider-generated types directly from the compiler so relocation and static linking no longer rely on provider-produced assemblies.

## Authoring a provider

1. **Static parameter**: expose a `ProvidedStaticParameter("Source", typeof<System.Type>)` that expects a non-generic, non-anonymous source type. When the compiler applies static arguments it will project the incoming `System.Type` into a TastReflection proxy that still reports the real metadata (attributes, optional flags, indexers, etc.).
2. **Generate a concrete type**: create a `ProvidedTypeDefinition` with `isErased = false`. Each member you add must also be non-erased.
3. **Emit members**: continue to add methods/properties/events via `ProvidedMethod`, `ProvidedProperty`, etc. The compiler captures the invoker expression or IL quotation you supply, so keep those expressions self-contained (no direct references back to the provider assembly).
4. **Definition locations**: call `AddXmlDocDelayed`/`AddDefinitionLocation` so the compiler can surface accurate diagnostics in the generated members. These are preserved inside `ProvidedMemberBinding` and flow into IDE tooling.

### Tooling / FCS helpers

- **`FSharpEntity.GetTypeReflectionProxy()`** (new in this branch) lets tooling authors retrieve the TastReflection `System.Type` proxy for any entity surfaced through `FSharp.Compiler.Service`. This removes the need for custom reflection hacks when a workspace wants to re-project the generated surface—simply grab the entity from `AssemblyContents`/`FSharpSymbolUse` and call the helper.
- **`ImportMap.ReflectTypeWithDependencies`** continues to serve low-level scenarios where you already have a `TyconRef`/`TType`. It now records every `TyconRef` touched so incremental rebuilds have accurate invalidation data.
- **SDK round-trip tests** (`FSharp.TypeProviders.SDK/tests/ProxyTypeTests.fs`) exercise both helpers: they compile a sample provider, fetch the proxy through `FSharpEntity.GetTypeReflectionProxy()`, and assert that `ProvidedTypesContext.ConvertTargetTypeToSource` sees the same type the compiler projected. Provider/test authors can use the same pattern to validate their own scenarios.

### Version pairing & rollout guidance

- **Compiler bits:** FS-1023 relies on the compiler artifacts produced from the `fs-1023` branch (or any nightly that postdates `FSharp.Core` **9.0.303**). Earlier toolchains do not surface `ProvidedMemberBinding` bodies and will fall back to provider-emitted IL.
- **SDK bits:** Provider authors should consume the matching TPSDK snapshot (the same revision that adds TastReflection proxy detection). Mixing the new compiler with an older SDK package reintroduces relocation issues because the SDK cannot recognise the compiler’s proxy types.
- **Documentation for consumers:** when you publish a provider that depends on FS-1023, state the minimum compiler/SDK pairing (e.g., “F# 9.0 preview 3 + TPSDK nightly 2025-11-12”). The feature remains behind `--langversion:preview` until GA.
- **Tooling:** ensure `FSharp.Compiler.Service`, `FSharp.Core`, and the TPSDK packages come from the same build. Editors or analyzers that mix versions will miss helpers such as `FSharpEntity.GetTypeReflectionProxy()`.

### Authoring checklist

When you adopt FS-1023, keep the following rules in mind:

1. **Trust the TastReflection proxies.** The compiler now preserves optional arguments, indexer shapes, and attribute data when it hands you the source `System.Type`, so prefer the proxy over reflection on the raw runtime type.
2. **Keep invoker quotations self-contained.** The compiler captures the quotations you hand to `ProvidedMethod`/`ProvidedProperty` and emits IL from them. Avoid calling back into the design-time assembly from those quotations or you’ll reintroduce the relocation issues FS-1023 is meant to solve.
3. **Use `ProvidedTypesContext.ConvertTargetTypeToSource`.** If you need to map a TastReflection proxy back to the user-authored `System.Type` (for caching or comparison), use the SDK helper rather than comparing raw `System.Type` instances.
4. **Don’t re-register relocated types.** The compiler registers the provider path (`Fs1023/Provided`) and the consumer path (`Fs1023Consumer/Provided`) inside `ProvidedGeneratedTypeRegistry`. If you add additional entries you can end up with duplicate IL type definitions.

### Restrictions

- **Anonymous records and type parameters** are rejected as static arguments. The compiler issues diagnostics (`FS0030`-style) encouraging users to pass a concrete, named type instead.
- **Provided types as static arguments** remain unsupported; FS-1023 only accepts `System.Type` instances that originate from user code or referenced assemblies.
- **Non-generated provider types** (e.g., erased types or types that reuse project-defined symbols) are blocked. If a provider tries to hand back a type that cannot be relocated, the compiler now surfaces “type could not be found in that assembly.”
- **Generics**: when reflecting over a generic source type (e.g., `Generic<'T>`), TastReflection scopes type parameters so the provider sees the original names/kinds, but the static argument itself must still be a constructed type (`Generic<int>`, `Generic<string>`, …) at the call site.

## What the compiler now guarantees

- **Dependency tracking**: `ImportMap.ReflectTypeWithDependencies` records every `TyconRef` touched while projecting the static argument. Incremental builds use this dependency set to invalidate providers when the source type changes.
- **Member publication**: `publishProvidedMembers` synthesises `Val` entries for every provided member, attaches the captured invoker expression, and registers them inside `ProvidedGeneratedTypeRegistry`. `tcaug_adhoc_list` therefore represents the entire generated surface.
- **IL emission**: `GenProvidedTypeDef` consumes those published members, emits real IL methods/constructors, and projects the collected property metadata into `ILPropertyDef` records. Registering a generated tycon now flips `SkipProviderStaticLinking`, so the static-linker skips embedding the provider DLL and `/standalone` builds reuse the compiler-emitted IL.
- **Reflection**: `TastReflection` and `TypeReflectionBuilder` mirror the final IL surface. IDE tools (FCS, FSC IntelliSense) can inspect parameter metadata (optional/default values, `ParamArray`, etc.) without talking to the provider again.

## Recommended testing workflow

1. **Compiler tests**: add or expand scenarios in `tests/FSharp.Compiler.Service.Tests/TypeProviderDependencyInvalidationTests.fs`. The existing suite covers records, unions, generics, and `/standalone` relocation by reflecting over the compiled consumer assembly.
2. **IDE-style smoke tests**: run `dotnet test tests/FSharp/FSharpSuite.Tests.fsproj -c Release --filter "FullyQualifiedName~TypeProviderTests"`. This executes the historical hello-world provider suite (FSC/FSI/bincompat) against the new pipeline.
3. **C# consumer**: compile a small C# console app that references the generated consumer DLL and asserts the static properties surface as expected. This ensures interop scenarios do not regress.
4. **Standalone builds**: pass `--standalone` (or set `FSharpStandalone=true` via MSBuild) to confirm that IlxGen-emitted IL survives relocation without needing the original provider assembly.

## Telemetry & diagnostics

FS-1023 ships with two complementary diagnostics channels:

- **Structured EventSource** – the compiler publishes events under the `FSharpCompiler-FS1023` provider. Enable it via the new command-line flag `--fs1023telemetry:<enable|disable|auto>` (or the MSBuild property `<Fs1023Telemetry>enable</Fs1023Telemetry>` on your project). IDE hosts can call `FSharpChecker.Create(enableFs1023Telemetry = true)` to flip the same switch. Once enabled, capture traces with e.g. `dotnet-trace collect --process-id <pid> --providers FSharpCompiler-FS1023` and inspect informational events such as `ProvidedTypeEmitted` (payload: `typeName`, `methodCount`, `propertyCount`, `eventCount`).
- **File-backed logs** – `FS1023_TRACE=1` still routes detailed `[fs1023][*]` entries into `/tmp/fs1023_trace.log` (override via `FS1023_TRACE_PATH`). This is useful for sharing repro artifacts or when EventSource tooling is unavailable.

Recommended workflow when escalations arrive:

1. Set `Fs1023Telemetry=enable` in the failing project (or pass `--fs1023telemetry:enable` on CLI invocations) and re-run the reproducible `dotnet test`/`dotnet build` command under `dotnet-trace collect`. The resulting trace will contain `ProvidedTypeEmitted` events for every generated type plus the wall-clock sequence of compiler phases.
2. If you need the historical textual breadcrumbs, also export `FS1023_TRACE=1 FS1023_TRACE_PATH=$(pwd)/fs1023.log`; the same action now automatically duplicates each line into the EventSource so you can correlate both feeds by timestamp.
3. When sharing diagnostics externally, bundle the `.nettrace` file, any `/tmp/fs1023-*` directories printed by the regression harness, and note the telemetry mode you used (`auto`, `enable`, `disable`).

## Release notes (preview draft)

**Feature scope**
- Type providers can take `System.Type` static arguments; the compiler projects the source shape via TastReflection and records every dependency so edits invalidate downstream consumers automatically.
- Generated members (`ProvidedMemberBinding`) live entirely inside the typed tree; IlxGen uses those bindings to emit IL, so `/standalone` builds and static linking no longer depend on provider-supplied assemblies.
- TastReflection + TypeReflectionBuilder proxies surface the generated surface to FCS/IDE tooling, with helper APIs (`FSharpEntity.GetTypeReflectionProxy`, `TcImports.GetTypeReflectionBuilder`) marked experimental for the preview.
- Observability now includes structured telemetry (`--fs1023telemetry:<enable|disable|auto>` / `<Fs1023Telemetry>` / `FSharpChecker.Create(enableFs1023Telemetry = true)`) plus the legacy `FS1023_TRACE` logs.

**Compatibility / requirements**
- Feature remains behind `--langversion:preview` and requires the matching TPSDK snapshot (documented earlier in this file). Mixing the updated compiler with older SDK packages will reintroduce relocation issues.
- The compiler emits preview-only APIs; IntelliSense warns consumers via `[<Experimental("FS-1023 preview API. Subject to change.")>]` attributes.

**Validation snapshot**
- `timeout 600s dotnet test tests/FSharp.Compiler.Service.Tests/FSharp.Compiler.Service.Tests.fsproj -c Release --filter "FullyQualifiedName~TypeProviderDependencyInvalidationTests" -- RunConfiguration.DisableParallelization=true`
- `timeout 600s dotnet test tests/FSharp.Compiler.ComponentTests/FSharp.Compiler.ComponentTests.fsproj -c Release`
- `timeout 300s dotnet test tests/FSharp.Compiler.Service.Tests/FSharp.Compiler.Service.Tests.fsproj -c Release --filter "DisplayName~mutable provided type exposes setter and constructor"`
- `timeout 300s dotnet test tests/FSharp.Compiler.Service.Tests/FSharp.Compiler.Service.Tests.fsproj -c Release --filter "DisplayName~provided event surfaces in emitted IL"`

**Known issues / limitations**
- Type-argument sources must be concrete, non-provided types; anonymous records and SRTP-heavy shapes are still blocked.
- Provider chaining (passing a provided type as the static argument to another provider) remains out of scope for the preview.
- IDE telemetry must be opted into explicitly until the feature graduates from preview status.

## Guidance for users

- Use `System.Type` arguments that refer to **stable public shapes**; private or internal nested types may change without notice and will trigger incremental rebuilds every time.
- Treat FS-1023 as an **incremental compiler feature**: the metadata for the static argument is recorded, so edits to the source type will invalidate downstream consumers just like edits to a referenced assembly.
- Continue to distribute your provider assembly; FS-1023 does not eliminate the need for the design-time DLL, but it does mean the final consumer IL no longer calls back into that DLL at runtime.
